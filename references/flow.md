# 开发流程

> 统一开发流程，适用于所有项目

---

## 标准流程（唯一流程）

```
需求收集 → 需求文档 + UI设计 + API接口定义 → 前后端并行开发（含后台管理系统）+ 联调 → 测试 + 代码质量检查 → 部署到测试服务器 → 完成
```

| 阶段 | 输出 | 门禁 | 谁参与 |
|------|------|------|--------|
| 需求收集 | 需求讨论（逐个问题） | ubilCore功能确定 | PM + 用户 |
| 需求文档 + UI设计 + API定义 | PRD文档 + UI设计方案 + API接口文档 | 用户确认 | PM + UI Designer + Backend |
| 前后端并行开发 | 可运行代码（用户端 + 后台管理系统） | CI 测试全过 | Backend + Frontend + DevOps |
| 联调 | 前后端对接完成（含后台管理系统对接） | 联调通过 | Backend + Frontend |
| 测试 + 代码质量 | 测试报告 + 质量报告 | QA 确认 | QA + DevOps |
| 部署到测试服务器 | 测试服务器可访问 | 部署成功可访问 | DevOps |
| 完成 | 交付 | - | - |

---

## 阶段详细说明

### 阶段1：需求收集

**目标**：明确项目要做什么

**输出**：核心功能清单

**规则**：
- 使用 `references/requirements-questionnaire.md` 模板
- 逐个问题询问用户，不要一次性问完
- 拿到足够信息就继续下一个阶段
- 环境问题自己解决，不问用户

**问题模板包含**：
- 项目基本信息（背景、用户、平台）
- 用户角色与权限
- 核心功能与优先级
- 数据与内容
- 第三方集成
- 非功能需求（含多语言**默认开启**、深色主题**默认开启**）
- 上线与验收标准

**默认开启项**：多语言支持、深色主题，除非用户明确说不要

### 阶段2：需求文档 + UI设计 + API定义

**目标**：产出 PRD、UI 设计稿和 API 接口文档

**输出**：
- `projects/<项目名>/PRD.md` - 产品需求文档
- `projects/<项目名>/UI设计.md` - UI 设计文档
- `projects/<项目名>/API.md` - API 接口文档

**规则**：
- 必须等用户确认 PRD、UI 设计和 API 文档后才能进入开发
- **API 接口文档是前后端并行开发的前提**
- Backend Agent 负责数据库设计和 API 接口定义
- UI Designer 负责 UI 设计
- PM 负责 PRD

### 阶段3：前后端并行开发（含后台管理系统）

**目标**：用户端和后台管理系统并行开发，互不依赖

**前提**：必须先完成 API 接口文档定义

**规则**：
- **环境自己搭建**：后端在本地或测试服务器搭建开发环境，前端在本地搭建
- **使用 Mock 数据**：前端使用 Mock 数据开发，基于 API 文档模拟接口
- **同步开发**：用户端和后台管理系统并行开发，互不阻塞
- **后台管理系统必须开发**：每个产品都要有配套的后台管理系统，方便运维
- **环境搭建规则**：
  - 本地能搞就在本地搞
  - 本地搞不了就在测试服务器（120.27.202.25）搭建
  - 数据库、中间件等服务使用 Docker 部署

**Backend Agent 职责**：
- 数据库设计（Schema、索引、迁移脚本）
- API 接口实现
- 业务逻辑开发

**后台管理系统功能参考**（根据产品类型选择）：
| 功能 | 说明 |
|------|------|
| 用户管理 | 查看用户列表、禁用/启用用户 |
| 数据统计 | 运营数据、订单数据、用户增长 |
| 内容管理 | 内容审核、发布管理 |
| 系统配置 | 开关配置、参数设置 |
| 日志查看 | 操作日志、错误日志 |
| 运维工具 | 缓存管理、任务调度 |

### 阶段4：联调

**目标**：前后端对接，接口互通

**规则**：
- 后端提供真实 API
- 前端对接真实接口
- 前后端共同解决联调问题
- 部署到测试服务器验证

### 阶段5：测试 + 代码质量检查

**目标**：确保代码质量和功能正确

**输出**：
- 测试报告
- 代码质量报告

**规则**：
- 必须运行测试用例
- 代码质量检查（可使用工具）
- 修复发现的问题

### 阶段6：部署到测试服务器

**目标**：将项目部署到测试环境供验证

**输出**：
- 测试服务器可访问的运行服务

**规则**：
- **必须使用 Docker 部署**（使用 docker-compose）
- 部署完成后**必须验证服务可访问**
- 测试服务器地址：120.27.202.25

---

## 测试环境 Docker 部署规范

**测试服务器**：120.27.202.25（root）

**部署规则**：
1. 前后端项目**必须使用 Docker 部署**
2. Backend：使用多阶段构建 Dockerfile
3. Frontend：使用 nginx Alpine 镜像
4. 使用 `docker-compose.yml` 统一编排
5. 部署完成后必须验证服务可访问

## 环境搭建规则（自己搞定）

| 环境 | 搭建位置 | 说明 |
|------|---------|------|
| 前端开发环境 | 本地 | Node.js + npm |
| 后端开发环境 | 本地或测试服务器 | Go + MySQL |
| 数据库 | 测试服务器 Docker | MySQL/PostgreSQL |
| Redis | 测试服务器 Docker | 缓存服务 |
| 前端预览 | 测试服务器 Docker | nginx |

## CI Gate 强制规则

### 每次 Commit 必须：

```
1. 自动运行测试
2. 判断结果

if PASS:
  允许提交
else:
  阻断流程 + 修复
```
