# 项目管理规范

> 标准项目管理模块，涵盖任务拆解、进度追踪、验收标准、风险管理和变更管理。

---

## 1. 任务拆解与看板

每个项目必须有任务看板，使用 `tasks/` 目录管理。

### 目录结构

```
projects/<项目名>/
├── tasks/
│   ├── backlog/          # 待办任务
│   ├── in_progress/      # 进行中
│   ├── done/            # 已完成
│   └── blocked/          # 被阻塞
└── kanban.md            # 看板总览
```

### 任务结构（tasks.json）

```json
{
  "id": "task_001",
  "title": "实现用户登录功能",
  "description": "包括手机号+验证码登录",
  "assigned_to": "backend",
  "status": "todo",
  "priority": "high",
  "story_points": 3,
  "acceptance_criteria": [
    "手机号格式校验",
    "验证码6位数字",
    "登录成功返回token",
    "登录失败返回错误码"
  ],
  "depends_on": ["task_000"],
  "created_at": "2026-04-03T01:00:00Z",
  "started_at": null,
  "completed_at": null
}
```

### 任务状态

| 状态 | 说明 |
|------|------|
| `todo` | 待办 |
| `in_progress` | 进行中 |
| `done` | 已完成 |
| `blocked` | 被阻塞（需要其他任务完成） |
| `cancelled` | 已取消 |

### 优先级

| 优先级 | 说明 | 建议处理时间 |
|--------|------|--------------|
| `critical` | 必须马上做 | 当天 |
| `high` | 重要 | 3天内 |
| `medium` | 一般 | 1周内 |
| `low` | 可以等 | 后续迭代 |

## 2. 验收标准（Acceptance Criteria）

每个任务必须有明确的验收标准。

### 验收标准格式

```markdown
## 验收标准（AC）

### AC1: 功能可用
- [ ] 用户可以输入手机号登录
- [ ] 验证码为6位数字
- [ ] 登录成功跳转首页

### AC2: 错误处理
- [ ] 手机号格式错误提示"手机号格式不正确"
- [ ] 验证码错误提示"验证码错误"
- [ ] 网络错误提示"网络异常"

### AC3: 性能
- [ ] 登录响应时间 < 2秒
```

### 验收标准规则

1. **每个任务必须有 AC**
2. **AC 必须具体可测试**
3. **完成后必须逐条核对**
4. **未完全满足 AC 的任务不能标记为 Done**

## 3. 里程碑（Milestones）

定义项目关键里程碑，明确交付时间。

### 里程碑结构

```json
{
  "id": "milestone_001",
  "title": "MVP 版本发布",
  "description": "包含核心功能的第一个可用版本",
  "target_date": "2026-04-15",
  "tasks": ["task_001", "task_002", "task_003"],
  "status": "in_progress",
  "completed_at": null
}
```

### 标准里程碑

| 里程碑 | 说明 | 典型交付物 |
|--------|------|-----------|
| M1: 需求冻结 | 需求文档确认，不再变更 | PRD、UI设计、API文档 |
| M2: 功能完成 | 所有功能开发完成 | 可运行的代码 |
| M3: 测试通过 | 核心功能测试通过 | 测试报告 |
| M4: 部署上线 | 部署到测试服务器 | 访问地址 |
| M5: 验收通过 | 用户验收通过 | UAT报告 |

## 4. 工作量估算

使用 Story Point 估算每个任务的工作量。

### 估算规则

| Story Point | 人天 | 说明 |
|-------------|------|------|
| 1 | 0.5天 | 非常简单，如改个文案 |
| 2 | 1天 | 简单，如加个字段 |
| 3 | 2天 | 中等，如开发一个小功能 |
| 5 | 3-5天 | 复杂，如带数据库的功能 |
| 8 | 5-7天 | 很复杂，如多个系统对接 |
| 13 | 1-2周 | 巨大功能块 |

### 估算流程

```
1. 任务拆解（Task）
2. 估算每个任务 Story Point
3. 汇总总 Story Point
4. 计算总人天 = 总SP / 2（单人开发速度）
5. 设定里程碑日期
```

### 速度基准

| 开发角色 | 速度（SP/人天） |
|----------|----------------|
| Backend | 2 SP/天 |
| Frontend | 2.5 SP/天 |
| UI Designer | 3 SP/天 |

## 5. 风险登记册

识别、评估和应对项目风险。

### 风险结构

```json
{
  "id": "risk_001",
  "title": "后端接口延期",
  "description": "后端开发进度比预期慢",
  "probability": "high",
  "impact": "high",
  "risk_score": 9,
  "mitigation": "前端先用Mock数据开发，不等待",
  "contingency": "增加后端人力",
  "status": "open",
  "owner": "backend",
  "created_at": "2026-04-03T01:00:00Z",
  "resolved_at": null
}
```

### 风险等级

| 分数 | 等级 | 说明 |
|------|------|------|
| 1-3 | 低 | 可接受风险，监控即可 |
| 4-6 | 中 | 需要采取措施 |
| 7-9 | 高 | 必须立即处理 |

### 常见风险清单

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 后端接口延期 | 中 | 高 | 前端用Mock开发 |
| 需求变更 | 高 | 中 | 评估影响再接受 |
| 第三方服务不可用 | 低 | 高 | 降级方案 |
| 技术难题 | 中 | 中 | 预留技术攻关时间 |
| 部署失败 | 低 | 高 | 回滚预案 |

## 6. 需求变更管理

当用户提出新需求或修改现有需求时，必须走变更流程。

### 变更流程

```
用户提出变更
    ↓
评估影响（时间/成本/范围）
    ↓
变更委员会审批（用户+Orchestrator）
    ↓
接受 / 拒绝
    ↓
接受：更新任务、调整计划
```

### 变更评估表

```markdown
## 变更请求: XXX

**提出人**: 用户
**日期**: 2026-04-03

### 变更内容
> 描述变更内容...

### 影响评估
| 项目 | 影响 |
|------|------|
| 时间 | +2天 |
| 成本 | 0 |
| 范围 | 增加XXX功能 |
| 质量 | 可能影响 |
| 风险 | 新风险XXX |

### 决策
- [ ] 接受
- [ ] 拒绝
- [ ] 放入后续迭代

**审批人**: 
**日期**:
```

### 变更规则

1. **评估影响**后再决定是否接受
2. **已确认的 PRD** 变更需要用户明确同意
3. **小型变更**（<1人天）可由 Orchestrator 直接决定
4. **所有变更**必须记录到 `CHANGELOG.md`

## 7. 每日站会（Daily Standup）

虽然 Agent 不需要开会，但需要记录每日进度。

### 每日报告格式

```markdown
## 2026-04-03 进度报告

### Backend Agent
- **已完成**: task_001 用户登录API
- **进行中**: task_002 用户下单API
- **阻塞**: 无
- **明日计划**: task_002 下单API

### Frontend Agent
- **已完成**: task_010 登录页面
- **进行中**: task_011 首页开发
- **阻塞**: task_011 等待 task_002 API
- **明日计划**: task_011 首页 + 对接登录API

### QA Agent
- **已完成**: -
- **进行中**: 准备测试用例
- **阻塞**: 等待联调
- **明日计划**: 继续准备测试用例
```

### 进度更新频率

| 阶段 | 更新频率 |
|------|----------|
| 开发阶段 | 每天更新 state.json |
| 联调阶段 | 每完成一个接口更新 |
| 测试阶段 | 每完成一个测试更新 |

## 8. Definition of Done（DoD）

任务完成的判断标准。

### 通用 DoD

- [ ] 代码编写完成
- [ ] 单元测试通过（覆盖率 > 70%）
- [ ] 代码审查通过（Code Review）
- [ ] 验收标准（AC）全部满足
- [ ] 部署到测试环境验证通过
- [ ] 文档更新（如有 API 变更）

### Backend 额外 DoD

- [ ] API 文档已更新
- [ ] 数据库迁移脚本已编写
- [ ] 接口符合 RESTful 规范
- [ ] 错误码已定义

### Frontend 额外 DoD

- [ ] 响应式布局测试通过
- [ ] 暗黑模式支持（如需要）
- [ ] 多语言支持（如需要）

## 10. 项目交付清单

项目完成时的交付物清单。

### 交付物清单

| 交付物 | 说明 | 状态 |
|--------|------|------|
| 源代码 | 完整源码仓库 | ☐ |
| API 文档 | 接口定义文档 | ☐ |
| 部署手册 | 部署步骤文档 | ☐ |
| 操作手册 | 用户使用说明 | ☐ |
| 测试报告 | 测试用例和结果 | ☐ |
| 数据库字典 | 表结构说明 | ☐ |
| 环境配置 | .env 示例文件 | ☐ |

### 交付检查

```markdown
## 交付检查

### 代码检查
- [ ] 代码已提交到仓库
- [ ] 无敏感信息泄露
- [ ] 代码注释完整

### 文档检查
- [ ] README.md 已编写
- [ ] API 文档已更新
- [ ] 部署手册已编写（使用 Docker）

### 部署检查
- [ ] Docker 镜像已构建
- [ ] 测试服务器可访问
- [ ] 功能验证通过

### 验收检查
- [ ] 用户已验收
- [ ] 遗留问题已记录
- [ ] 交付清单已确认
```
