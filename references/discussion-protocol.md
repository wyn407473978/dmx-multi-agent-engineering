# 讨论决策协议 V2

## 核心原则

**逐个提问 + 即时执行** — 每次只问一个问题，拿到答案立即执行可执行的部分，不要一次性问完所有问题。

## 原协议的问题

❌ 旧做法：一次性问完所有问题 → 等所有答案 → 再讨论 → 再设计 → 再开发
✅ 新做法：**逐个提问 → 立即执行可执行的部分 → 并行推进**

## 新讨论流程

```
用户需求 → 提问 #1 → 拿到答案 → 立即执行可执行的部分
                        ↓
              提问 #2 → 拿到答案 → 继续推进
                        ↓
              （不等所有问题问完，相关任务已并行开始）
```

### 关键规则

| 规则 | 说明 |
|------|------|
| **一次只问一个问题** | 不要一次性抛出所有问题 |
| **拿到答案立即行动** | 有足够信息就启动相关 Agent |
| **并行执行** | 可以独立的任务同时进行 |
| **阻塞时继续推进** | 某个问题卡住时，先推进其他能确定的 |
| **核心确定就推进** | 核心功能确定后即可进入 DESIGN |

## 输出格式要求

**重要：所有文档输出必须使用标准 Markdown 格式**

| 文档类型 | 必须包含的Markdown元素 |
|---------|----------------------|
| PRD文档 | # 标题、## 章节、- 列表、| 表格、``` 代码块 |
| API文档 | # 标题、## 接口分组、### 接口名、``` json 示例 |
| 设计文档 | # 标题、## 模块、### 表格、``` 架构图 |
| 任务清单 | ## 标题、| 表格、- 列表 |

**不允许**：纯文本输出、缺少标题层级、不使用表格展示结构化数据

## 文档输出规则

**重要：文档不要在聊天中显示，直接上传到 GitHub**

| 操作 | 说明 |
|------|------|
| 文档输出 | 生成 .md 文件后直接上传到 GitHub 仓库 |
| 用户通知 | 告诉用户文档在 GitHub 的路径/地址 |
| 聊天消息 | 只说"文档已生成，地址：https://github.com/xxx" |

**原因**：聊天中 Markdown 显示不完整、无法滚动，用户体验差。

```

## 并行执行规则

### 可以立即并行的场景

1. **后端 API 和 前端 UI** 可以并行开发（接口约定好即可）
2. **PRD 和 UI 设计** 可以并行（PM写需求，UI开始画稿）
3. **技术选型和项目结构搭建** 可以并行

### 必须串行的场景

1. **接口定义** 必须先于前后端联调
2. **数据库 Schema** 必须先于后端开发
3. **页面结构** 必须先于详细 UI 开发

## 阶段演进规则

```
INIT → DISCUSSION → DESIGN → PRD确认 → DEV → ...
         ↓                  ↓
    核心功能确定 →        用户确认PRD后
    DESIGN开始            才能进入DEV
```

### PRD 确认规则（强制）

**DESIGN 阶段完成后，必须生成完整 PRD 文档，用户确认后才能进入开发阶段**

| 规则 | 说明 |
|------|------|
| **PRD 生成时机** | DISCUSSION + DESIGN 完成后 |
| **PRD 保存路径** | `projects/<项目名>/PRD.md` |
| **用户确认内容** | 功能列表、页面结构、API 接口、技术方案 |
| **确认方式** | 用户阅读 PRD 后明确说"确认" |
| **未确认后果** | 未确认不进入 DEV 阶段 |

### PRD 文档必须包含

| 章节 | 内容 |
|------|------|
| 项目概述 | 项目背景、目标、范围 |
| 用户角色 | 角色定义、功能权限 |
| 功能列表 | 所有功能点及描述 |
| 页面结构 | 页面清单、跳转关系 |
| API 接口 | 接口清单、请求/响应示例 |
| 技术方案 | 技术栈、数据库设计 |
| 非功能需求 | 性能、安全、兼容性 |

## Orchestrator 职责更新

1. **识别可执行任务** — 拿到部分信息就判断能做什么
2. **分发任务** — 不等所有Agent，集中分配
3. **跟踪进度** — 用 state.json 记录各任务状态
4. **处理阻塞** — 阻塞的任务挂起，先做其他的

## 执行日志规则

**重要：每个项目必须记录所有 Agent 的执行日志**

| 规则 | 说明 |
|------|------|
| **新建项目时** | 创建 `PROJECT_LOG.md` 作为执行日志 |
| **Agent 启动时** | 记录开始时间、任务描述、预期产出 |
| **Agent 完成时** | 记录完成时间、实际产出、状态 |
| **Agent 超时/失败** | 记录问题原因、已完成的产出 |

### 日志格式

```markdown
## [Agent名称] - 任务描述

| 字段 | 值 |
|------|-----|
| 任务 | <任务描述> |
| 开始时间 | <YYYY-MM-DD HH:mm> |
| 完成时间 | <YYYY-MM-DD HH:mm> |
| 状态 | ✅完成 / ❌失败 / ⏳进行中 |
| 产出 | <文件/目录/结果> |
| 备注 | <问题/备注> |
```

### 示例

```markdown
## frontend - 初始化前端项目

| 字段 | 值 |
|------|-----|
| 任务 | 初始化React项目，配置Ant Design |
| 开始时间 | 2026-04-02 10:30 |
| 完成时间 | 2026-04-02 10:35 |
| 状态 | ✅完成 |
| 产出 | frontend/ 目录 |
| 备注 | - |
```

### GitHub 上传

- 日志文件随项目一起 push 到 GitHub
- 用户可直接在 GitHub 查看执行历史
- 日志路径：`projects/<项目名>/PROJECT_LOG.md`

## 状态更新频率

- 每个有意义的信息获取 → 更新 state.json
- 每个任务完成 → 更新 state.json
- 不要等到"阶段完成"才更新
