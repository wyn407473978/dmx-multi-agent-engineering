# 讨论决策协议 V2

## 核心原则

**迭代式讨论 + 即时执行** — 不要等所有问题问完才动手。拿到什么信息就先开始，能并行就并行。

## 原协议的问题

❌ 旧做法：一次性问完所有问题 → 等所有答案 → 再讨论 → 再设计 → 再开发
✅ 新做法：**逐个提问 → 立即执行可执行的部分 → 并行推进**

## 新讨论流程

```
用户需求 → 逐个问题 → 拿到答案 → 相关Agent立即开始执行
                          ↓
              同时进行多个任务，不用等
```

### 关键变化

| 旧做法 | 新做法 |
|--------|--------|
| 问完所有问题再动手 | 拿到足够信息就动手 |
| 所有Agent同步讨论 | 相关Agent各自并行执行 |
| 达成完整共识才推进 | 核心决策确定就推进 |
| 阻塞是问题 | 阻塞挂起，继续其他任务 |

## 输出格式要求

**重要：所有文档输出必须使用标准 Markdown 格式**

| 文档类型 | 必须包含的Markdown元素 |
|---------|----------------------|
| PRD文档 | # 标题、## 章节、- 列表、| 表格、``` 代码块 |
| API文档 | # 标题、## 接口分组、### 接口名、``` json 示例 |
| 设计文档 | # 标题、## 模块、### 表格、``` 架构图 |
| 任务清单 | ## 标题、| 表格、- 列表 |

**不允许**：纯文本输出、缺少标题层级、不使用表格展示结构化数据

---

## 文档输出规则

**重要：文档不要在聊天中显示，直接上传到 GitHub**

| 操作 | 说明 |
|------|------|
| 文档输出 | 生成 .md 文件后直接上传到 GitHub 仓库 |
| 用户通知 | 告诉用户文档在 GitHub 的路径/地址 |
| 聊天消息 | 只说"文档已生成，地址：https://github.com/xxx" |

**原因**：聊天中 Markdown 显示不完整、无法滚动，用户体验差。

## 讨论格式

### 单问题讨论

```
【Orchestrator】提问 #1

## 问题
<一个具体问题>

## 已确认（基于已有信息）
- <已确定的事实>

## 可执行的任务（立即开始）
✅ <任务A> → <Agent>
✅ <任务B> → <Agent>
```

### Agent 回应格式（精简版）

```
【Agent】回答 #1

## 回答
<直接回答问题>

## 已开始执行
- <任务>：<进度或产出>
```

## 并行执行规则

### 可以立即并行的场景

1. **后端 API 和 前端 UI** 可以并行开发（接口约定好即可）
2. **PRD 和 UI 设计** 可以并行（PM写需求，UI开始画稿）
3. **技术选型和项目结构搭建** 可以并行

### 必须串行的场景

1. **接口定义** 必须先于前后端联调
2. **数据库 Schema** 必须先于后端开发
3. **页面结构** 必须先于详细 UI 开发

## 阶段演进规则

```
INIT → DISCUSSION → DESIGN → DEV → ...
         ↓
    不用等"全部讨论完"才进入DESIGN
    核心功能确定 → DESIGN开始
    核心接口确定 → DEV开始
```

## Orchestrator 职责更新

1. **识别可执行任务** — 拿到部分信息就判断能做什么
2. **分发任务** — 不等所有Agent，集中分配
3. **跟踪进度** — 用 state.json 记录各任务状态
4. **处理阻塞** — 阻塞的任务挂起，先做其他的

## 执行日志规则

**重要：每个项目必须记录所有 Agent 的执行日志**

| 规则 | 说明 |
|------|------|
| **新建项目时** | 创建 `PROJECT_LOG.md` 作为执行日志 |
| **Agent 启动时** | 记录开始时间、任务描述、预期产出 |
| **Agent 完成时** | 记录完成时间、实际产出、状态 |
| **Agent 超时/失败** | 记录问题原因、已完成的产出 |

### 日志格式

```markdown
## [Agent名称] - 任务描述

| 字段 | 值 |
|------|-----|
| 任务 | <任务描述> |
| 开始时间 | <YYYY-MM-DD HH:mm> |
| 完成时间 | <YYYY-MM-DD HH:mm> |
| 状态 | ✅完成 / ❌失败 / ⏳进行中 |
| 产出 | <文件/目录/结果> |
| 备注 | <问题/备注> |
```

### 示例

```markdown
## frontend - 初始化前端项目

| 字段 | 值 |
|------|-----|
| 任务 | 初始化React项目，配置Ant Design |
| 开始时间 | 2026-04-02 10:30 |
| 完成时间 | 2026-04-02 10:35 |
| 状态 | ✅完成 |
| 产出 | frontend/ 目录 |
| 备注 | - |
```

### GitHub 上传

- 日志文件随项目一起 push 到 GitHub
- 用户可直接在 GitHub 查看执行历史
- 日志路径：`projects/<项目名>/PROJECT_LOG.md`

## 状态更新频率

- 每个有意义的信息获取 → 更新 state.json
- 每个任务完成 → 更新 state.json
- 不要等到"阶段完成"才更新

## 示例：宠物领养网站

```
【Orchestrator】提问 #1

## 问题
网站的核心功能是什么？领养流程需要哪些步骤？

## 已确认
- 平台：网页端（响应式）
- 用户：领养者、救助者、管理员

## 可执行任务（立即开始）
✅ 救助者发布宠物信息流程设计 → product_manager
✅ 宠物展示页面结构 → ui_designer
✅ 项目初始化 + 技术选型 → backend

---

【product_manager】回答 #1

## 回答
核心功能：
1. 宠物列表浏览（筛选：品种、地区、年龄）
2. 领养申请（填写信息 → 救助者审核 → 领养成功）
3. 救助者发布宠物

## 已开始执行
- 正在输出：领养流程详细步骤

---

【Orchestrator】立即分配

✅ product_manager → 输出领养流程文档
✅ backend → 开始搭建项目结构，准备API设计
✅ ui_designer → 开始宠物列表页面线框图

（不等其他问题答案，先跑起来）
```
